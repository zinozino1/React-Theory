1. 리덕스 기본 개념 정리

1) 액션 : 상태 업데이트를 할 때 참고해야할 값. type 프로퍼티가 필수적으로 있어야함 나머지 값은 작성자 마음대로 넣을 수 있다.

2) 액션 생성 함수 : 액션 객체를 만들어주는 함수.

3) 리듀서 : 변화를 일으키는 함수. 액션 객체를 받아서 새로운 상태를 만들어 반환

3) 스토어 : 현재 애플리케이션 상태와 리듀서가 들어있고 중요한 내장함수를 지닌다. 한 개의 프로젝트는 단 하나의 스토어만 가질 수 있음

4) 디스패치 : 스토어의 내장 함수 중 하나로 액션트리거 라고 생각하면 됨

5) 구독 : 스토어의 내장 함수 중 하나로 이 함수 안에 리스너 함수를 파라미터로 넣어서 호출해주면 이 리스너 함수가 액션이 디스패치 되어 상태가 업데이트될 때마다 호출된다. 보통 react-redux라이브러리를 사용한다.

**** useReducer vs redux

useReducer
- state는 각 컴포넌트 별로 관리
- state 변화 logic은 컴포넌트 외부에서
- action 개발자가 일일이 정의

redux
- state는 store에서 따로 관리
- state 변화 logic인 reducer store에서 관리
- action 생성 함수가 존재




2. 리덕스의 세 가지 규칙

1) 단일 스토어
- 하나의 어플리케이션에는 하나의 스토어만 만들 것을 권장

2) 읽기 전용 상태
- 상태를 업데이트 할 때는 기존의 객체는 건드리지 않고 새로운 객체를 생성해주어야 함

3) 리듀서는 순수한 함수
- 리듀서 함수는 이전 상태와 액션 객체를 파라미터로 받는다
- 파라미터 외의 값에는 의존하면 안된다.
- 이전 상태는 절대 건드리지 않고, 변화를 준 새로운 상태 객체를 만들어서 반환
- 똑같은 파라미터로 호출된 리듀서 함수는 언제나 똑같은 결과 값을 반환해야함

* 예를 들어 리듀서 함수 내에서 랜덤 값을 만들거나 date함수 사용, 네트워크 요청을 하면 파라미터가 같아도 다른 결과를 만들어 낼 수 있기 때문에 사용하면 안된다. 이러한 작업은 리듀서 바깥에서 처리해주어야 하는데 액션을 만드는 과정에서 처리해도 되고, 리덕스 미들웨어에서 처리해도 된다. 주로 네트워크 요청과 같은 비동기 작업은 미들웨어를 통해 관리한다.
